# Password Security — Deep Module

## Threat Description

Weak password handling is the most common authentication vulnerability. Improper storage, weak policies, and insecure reset flows lead to credential theft and account takeover.

---

## Password Hashing

### Algorithm Selection

```
RULE: Use a dedicated password hashing algorithm. NOT a general-purpose hash.

✓ APPROVED (in order of preference):
1. Argon2id — Winner of Password Hashing Competition (2015). Best choice.
2. bcrypt — Battle-tested, widely supported. Good choice.
3. scrypt — Memory-hard. Good but less common.

✗ BANNED for password hashing:
- MD5 (broken, fast, no salt)
- SHA-1 (broken, fast)
- SHA-256/SHA-512 (too fast — billions of guesses/second)
- PBKDF2-SHA1 (acceptable only if nothing else available, use high iterations)
- Plain text (obviously)
- Reversible encryption (passwords should never be decryptable)
- Base64 encoding (not encryption at all)
- Custom hashing schemes
```

### Argon2id Configuration

```
RECOMMENDED configuration:
- Variant: Argon2id (hybrid — resistant to side-channel and GPU attacks)
- Memory: 64 MB minimum (higher is better — 256 MB for high-security)
- Iterations: 3 minimum
- Parallelism: 1 (or number of CPU cores)
- Salt: 16 bytes random (auto-generated by libraries)
- Hash length: 32 bytes

// Node.js (argon2)
const argon2 = require('argon2');
const hash = await argon2.hash(password, {
  type: argon2.argon2id,
  memoryCost: 65536,   // 64 MB
  timeCost: 3,
  parallelism: 1
});
const valid = await argon2.verify(hash, password);

// Python (argon2-cffi)
from argon2 import PasswordHasher
ph = PasswordHasher(memory_cost=65536, time_cost=3, parallelism=1)
hash = ph.hash(password)
ph.verify(hash, password)

Tuning: Hash should take 0.5-1.0 seconds on your production hardware.
```

### bcrypt Configuration

```
RECOMMENDED configuration:
- Cost factor: 12 minimum (higher = slower = more secure)
- Salt: auto-generated (built into bcrypt)
- Max password length: 72 bytes (bcrypt limitation — truncates longer passwords)

// Node.js (bcrypt)
const bcrypt = require('bcrypt');
const hash = await bcrypt.hash(password, 12);  // 12 rounds
const valid = await bcrypt.compare(password, hash);

// Python (bcrypt)
import bcrypt
hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))
valid = bcrypt.checkpw(password.encode(), hash)

// Java (Spring Security)
BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(12);
String hash = encoder.encode(password);
boolean valid = encoder.matches(password, hash);

// C# (BCrypt.Net)
string hash = BCrypt.Net.BCrypt.HashPassword(password, workFactor: 12);
bool valid = BCrypt.Net.BCrypt.Verify(password, hash);

IMPORTANT: bcrypt truncates at 72 bytes. For longer passwords:
  - Pre-hash with SHA-256: bcrypt(SHA256(password))
  - Or use Argon2id (no length limit)
```

## Password Policy

### Requirements

```
Modern password policy (NIST SP 800-63B aligned):

REQUIRED:
- Minimum length: 8 characters (12+ recommended)
- Maximum length: 128+ characters (don't limit unnecessarily)
- Allow all printable ASCII characters + Unicode
- Allow spaces (passphrases)
- Check against breached password database (Have I Been Pwned API)
- Check against common password list (top 10,000 passwords)
- Check for context-specific weak passwords (username, email, site name)

NOT RECOMMENDED (NIST says don't):
- Complexity requirements (uppercase, lowercase, number, symbol)
  (Users create predictable patterns: Password1! → not more secure)
- Periodic password rotation (causes weaker passwords)
  (Only force change on breach evidence)
- Password hints (leak information)
- Security questions (easily researched)

WHY: Long passphrases ("correct horse battery staple") are more secure
     than short complex passwords ("P@ssw0rd1!")
```

### Breached Password Check

```
RULE: Check passwords against known breach databases.

Have I Been Pwned (HIBP) API — k-Anonymity model:
1. Hash password with SHA-1
2. Send first 5 characters of hash to API
3. API returns all matching hashes (suffix + count)
4. Check locally if full hash matches

// Node.js
const crypto = require('crypto');
const sha1 = crypto.createHash('sha1').update(password).digest('hex').toUpperCase();
const prefix = sha1.slice(0, 5);
const suffix = sha1.slice(5);
const response = await fetch(`https://api.pwnedpasswords.com/range/${prefix}`);
const breached = response.text().includes(suffix);

This is privacy-safe — full password hash never leaves your server.

RULE: Block passwords found in breaches.
RULE: Check on registration AND on password change.
RULE: Optionally check on login and prompt user to change if breached.
```

### Password Strength Meter

```
RECOMMENDED: Use zxcvbn (realistic strength estimation).

zxcvbn considers:
- Dictionary words
- Common passwords
- Keyboard patterns (qwerty, 123456)
- L33t speak substitutions
- Dates, names, repeated characters
- Spatial patterns

// Client-side feedback:
const result = zxcvbn(password);
// result.score: 0-4 (0 = worst, 4 = best)
// result.feedback.warning: human-readable warning
// result.feedback.suggestions: improvement tips

RULE: Require minimum score of 3 for password acceptance.
RULE: Show strength meter for UX but validate server-side.
```

## Password Reset

```
Secure password reset flow:

1. Request reset:
   - User submits email
   - ALWAYS respond: "If this email is registered, a reset link has been sent"
   - Rate limit: 3 requests per hour per email
   - Generate token: crypto.randomBytes(32).toString('hex')
   - Hash token before storing: SHA-256(token)
   - Set expiry: 15-30 minutes
   - Send reset link via email: https://example.com/reset?token=UNHASHED_TOKEN

2. Reset page:
   - Validate token (hash submitted token, compare with stored hash)
   - Check expiry
   - Enforce password policy on new password
   - Show password strength meter

3. After reset:
   - Invalidate the reset token
   - Hash and store new password
   - Invalidate ALL active sessions for the user
   - Send confirmation email ("Your password was changed")
   - Log the event (IP, timestamp, user agent)

4. Token security:
   - Single-use (delete after use or first attempt)
   - Cryptographically random (not sequential, not timestamp-based)
   - Hashed in database (like passwords — if DB is compromised, tokens useless)
   - Short expiry (15-30 minutes)
```

## Password Storage Upgrade

```
IF migrating from weak hashing (MD5, SHA-1) to strong hashing:

Strategy 1: Wrap existing hash (immediate):
  new_hash = bcrypt(existing_md5_hash)
  Store: { algorithm: "bcrypt(md5)", hash: new_hash }
  On login: verify bcrypt(md5(password)) against stored hash
  Then re-hash with bcrypt(password) directly and update

Strategy 2: Re-hash on login (gradual):
  On login: verify against old hash algorithm
  If valid: re-hash with bcrypt/argon2, update stored hash
  Track migration progress (% of users migrated)
  After sufficient time: force password reset for remaining users

RULE: Never store old and new hashes simultaneously (doubles attack surface).
RULE: Log migration progress.
RULE: Set deadline for forced reset of un-migrated accounts.
```
