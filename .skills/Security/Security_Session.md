# Security — Session Management

## Scope

This module covers session lifecycle security: creation, validation, storage, timeout, and destruction for both server-side sessions and token-based sessions.

## Dependencies

```
IF task also involves cookies:
    LOAD modules/Cookie.md
```

## Session Security Rules

### 1. Session ID Generation

```
RULE: Session IDs MUST be cryptographically random.

Requirements:
- Minimum 128 bits of entropy (32 hex characters / 22 base64 characters)
- Use CSPRNG (Cryptographically Secure Pseudo-Random Number Generator)
  - Node.js: crypto.randomBytes(32)
  - Python: secrets.token_urlsafe(32)
  - Java: SecureRandom
  - C#: RandomNumberGenerator
  - Go: crypto/rand
  - PHP: random_bytes(32)

NEVER:
- Use Math.random() or equivalent
- Use timestamp-based IDs
- Use sequential IDs
- Use user ID or username as part of session ID
- Use MD5/SHA1 of predictable data
```

### 2. Session Storage

```
Server-Side Sessions (Recommended):
- Store session data on server (Redis, database, memory store)
- Client holds only session ID (in cookie)
- Session data never exposed to client
- Easy to invalidate server-side

Token-Based Sessions (JWT):
- See modules/JWT.md for details
- Stateless but harder to revoke
- Use short expiry + refresh token pattern
- Store refresh tokens server-side for revocation

RULE: If using server-side sessions, use a dedicated session store.
      Do NOT store sessions in the application database if high throughput is expected.

Recommended stores:
- Redis (fast, TTL support, cluster support)
- Memcached (fast, simple)
- Database (acceptable for low-traffic apps)
```

### 3. Session Configuration

```
RULE: Configure session with security-first defaults.

Cookie settings for session ID:
- Secure: true (HTTPS only)
- HttpOnly: true (no JavaScript access)
- SameSite: Lax or Strict
- Path: / (or specific path)
- Domain: specific domain (not wildcard)
- Max-Age: match session timeout

Session timeout:
- Idle timeout: 15-30 minutes (web apps)
- Absolute timeout: 8-24 hours (force re-authentication)
- Remember me: 30 days max (separate persistent token)
- Sensitive apps (banking): 5-15 minutes idle timeout
```

### 4. Session Lifecycle

```
RULE: Manage session state transitions securely.

Creation:
- Generate new session ID on login
- Store creation timestamp
- Store user agent and IP (for anomaly detection)
- Set initial session state

Validation (every request):
1. Session exists in store?
2. Session not expired (idle + absolute)?
3. Session bound to same user?
4. User account still active?
5. (Optional) IP/User-Agent consistent?

Rotation:
- Regenerate session ID after authentication
- Regenerate session ID after privilege change
- Regenerate periodically (every 15-30 minutes)
- Delete old session ID immediately after rotation

Destruction:
- On explicit logout: delete from store + clear cookie
- On password change: destroy ALL sessions for the user
- On account deactivation: destroy ALL sessions
- On timeout: delete from store
- On suspicious activity: destroy ALL sessions + notify user
```

### 5. Session Fixation Prevention

```
RULE: Always regenerate session ID after authentication state changes.

Vulnerable pattern:
1. Attacker gets a valid session ID (from the site)
2. Attacker tricks user into using that session ID
3. User logs in with the attacker's session ID
4. Attacker now has authenticated session

Prevention:
- Generate new session ID on login (MANDATORY)
- Reject session IDs not generated by the server
- Bind session to user-agent (optional, breaks some legitimate flows)
- Use Strict SameSite cookies
```

### 6. Session Hijacking Prevention

```
RULE: Protect session tokens from theft and misuse.

Transport security:
- HTTPS everywhere (Secure cookie flag)
- HSTS header enabled
- No session ID in URL (ever)
- No session ID in logs

Client-side protection:
- HttpOnly cookies (no JavaScript access)
- SameSite cookies (CSRF protection)
- No session data in localStorage (accessible to XSS)

Server-side protection:
- Validate session on every request
- Detect anomalies (IP change, user-agent change)
- Rate limit requests per session
- Log and alert on impossible travel (login from 2 countries in 1 minute)
```

### 7. Concurrent Session Control

```
RULE: Define policy for concurrent sessions.

Options:
A. Allow unlimited concurrent sessions (least restrictive)
B. Limit concurrent sessions (e.g., max 5 devices)
C. Notify on new login from different device
D. Single session only (most restrictive — banking apps)

For option B-D:
- Track active sessions per user
- Provide "view active sessions" UI
- Allow user to revoke individual sessions
- Show device, IP, last active time
- Auto-revoke oldest session when limit exceeded
```

### 8. Remember Me

```
RULE: "Remember Me" is a separate long-lived token, NOT an extended session.

Implementation:
1. Generate random token (separate from session ID)
2. Hash token before storing in database
3. Associate with user ID and device fingerprint
4. Set cookie with 30-day expiry (Secure, HttpOnly, SameSite)
5. On return visit: validate token → create new session → rotate token
6. Invalidate remember-me token on password change
7. Provide UI to view and revoke remembered devices

NEVER:
- Just extend session timeout for "remember me"
- Store unhashed remember-me tokens
- Allow remember-me to bypass MFA (require MFA on new session)
```

### 9. Logout

```
RULE: Logout must be thorough and immediate.

Logout checklist:
1. Delete session from server-side store
2. Clear session cookie (set Max-Age=0)
3. Clear remember-me cookie and token
4. Clear any cached auth state
5. Redirect to login page
6. Prevent back-button access to authenticated pages (Cache-Control: no-store)

Additional logout features:
- "Logout from all devices" option
- Log the logout event (user ID, timestamp, device)
- POST request for logout (prevent CSRF logout)
```

### 10. Session Monitoring

```
RULE: Monitor sessions for security anomalies.

Track per session:
- Creation time
- Last activity time
- IP addresses used
- User agent string
- Geographic location (from IP)
- Request count
- Failed request count

Alert on:
- Session used from multiple IPs simultaneously
- Session used from multiple user agents
- Impossible travel (geographic anomaly)
- Unusual request rate
- Access to unexpected resources
- Session used after user-initiated logout
```
